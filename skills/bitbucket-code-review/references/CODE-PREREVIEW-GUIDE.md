## 🎯 核心理念

**AI 智能审查** - 不依赖固定脚本，而是由 AI 深度理解代码和规范，进行灵活智能的审查。

### ⭐ 审查优先级原则（必须遵守）

```
完整审查 = 编码规范检查 + 架构设计分析 + 职责清晰性检查
           ↓（第一步）      ↓（第二步）         ↓（第三步）
        基础质量问题      架构层次问题       设计模式问题
```

**重要规则**：
1. ✅ **先检查编码规范** - 这是代码质量的基础，不可跳过
2. ✅ **再分析架构设计** - 检查职责分离、层次划分、设计一致性
3. ⚠️ **两者同样重要** - 编码规范和架构设计缺一不可
4. ❌ **禁止只做架构分析** - 必须包含完整的编码规范检查

**检查清单**：
- [ ] 魔法值（中文字符串、数字常量、字符串常量）
- [ ] @Override 注解
- [ ] 空值检查
- [ ] 类型安全（类型转换、泛型使用）
- [ ] 命名规范
- [ ] 集合处理规范
- [ ] 职责清晰性（单一职责、层次分离）
- [ ] 方法复杂度（是否需要拆分）
- [ ] 架构一致性

## 🚀 快速开始

### 第一次使用

1. **确保必需文件存在**
   - `fetch_pr_diff.py` - PR 获取工具
   - `bitbucket_pr_reader.py` - Bitbucket API 封装
   - `CODE-STANDARDS.md` - 编码规范文档
   - `ERROR-CODE-STANDARDS.md` - 错误码规范文档

2. **配置认证信息**（必需）
   
   脚本需要 Bitbucket API Token 和用户名才能访问 PR 信息。有两种配置方式：

   **方式 1：使用环境变量（推荐）**
   ```bash
   # Windows PowerShell
   $env:BITBUCKET_API_TOKEN="your_api_token_here"
   $env:BITBUCKET_USERNAME="your.email@company.com"
   
   # Windows CMD
   set BITBUCKET_API_TOKEN=your_api_token_here
   set BITBUCKET_USERNAME=your.email@company.com
   
   # Linux/Mac
   export BITBUCKET_API_TOKEN="your_api_token_here"
   export BITBUCKET_USERNAME="your.email@company.com"
   ```

   **方式 2：使用命令行参数**
   ```bash
   python fetch_pr_diff.py <PR_URL> --api-token YOUR_TOKEN --username YOUR_EMAIL
   ```

   **如何获取 API Token**：
   - 访问 Bitbucket 设置: https://bitbucket.org/account/settings/app-passwords/
   - 或访问 Atlassian 账户设置创建 API Token
   - Token 格式通常以 `ATATT` 开头
   - 用户名通常是您的邮箱地址（如：user@company.com）

3. **执行审查**
   ```bash
   # 使用环境变量
   python fetch_pr_diff.py <PR_URL>
   
   # 或使用命令行参数
   python fetch_pr_diff.py <PR_URL> --api-token YOUR_TOKEN --username YOUR_EMAIL
   ```

### 典型使用场景

**用户说**: "请审查 https://bitbucket.org/your-workspace/your-repo/pull-requests/42"

**AI 应该做**:
1. 检查环境变量是否已设置（BITBUCKET_API_TOKEN 和 BITBUCKET_USERNAME）
2. 运行 `python fetch_pr_diff.py <PR_URL>`（如果环境变量已设置）
   或提示用户设置环境变量/使用命令行参数
3. 读取生成的 diff 和 info 文件
4. 读取编码规范和错误码规范
5. 进行深度分析
6. 生成审查报告

## ⚡ 审查步骤

### Step 1: 获取 PR 变更

```bash
# 使用环境变量（推荐）
python fetch_pr_diff.py <PR_URL>

# 或使用命令行参数
python fetch_pr_diff.py <PR_URL> --api-token YOUR_TOKEN --username YOUR_EMAIL
```

**认证说明**：
- 脚本需要 Bitbucket API Token 和用户名
- 优先使用命令行参数，其次使用环境变量
- 如果两者都未提供，脚本会提示错误信息

**输出文件**：
脚本会自动创建带时间戳的文件夹，格式为：`<仓库名>_PR<PR_ID>_<时间戳>/`

时间戳格式：`YYYYMMDDHHmm`（年月日时分）

示例：`sp-search-citsgbt_PR174_202512261133/`

文件夹内包含：
- `<仓库名>_<PR_ID>_<时间戳>_diff.txt` - 代码变更
- `<仓库名>_<PR_ID>_<时间戳>_info.json` - PR 信息
- `<仓库名>_<PR_ID>_<时间戳>_review.md` - AI 生成的审查报告（由 AI 创建）

### Step 2: AI 进行智能审查

**AI 需要做的事情**：

1. 读取规范文档（每次都读取，确保使用最新规范）
   - `CODE-STANDARDS.md`
   - `ERROR-CODE-STANDARDS.md`

2. 读取 PR 内容（在带时间戳的文件夹中）
   - `<仓库名>_PR<PR_ID>_<时间戳>/<仓库名>_<PR_ID>_<时间戳>_diff.txt`
   - `<仓库名>_PR<PR_ID>_<时间戳>/<仓库名>_<PR_ID>_<时间戳>_info.json`

3. 深度分析代码（按顺序执行）
   
   **⚠️ 审查前必读**：
   - 仔细阅读 diff 文件中的实际新增代码（以 `+` 开头的行）
   - 基于实际代码内容进行审查，不要凭假设或猜测
   - 确认问题真实存在后再报告
   - 对于复杂情况，反复确认以避免误判
   
   **第一步：编码规范检查（基础必做）**
   - ✅ 魔法值检查
     - 数字常量（如 9999、100）
     - 字符串常量（如 "SABRE"、"无房过滤"、"不符合直连优先"）
     - 中文字符串必须定义为常量
   - ✅ OOP 规约检查
     - 所有覆写方法必须有 @Override 注解
     - **注意**：查看 diff 文件确认 @Override 是否真的缺失，不要误报
     - equals() 调用规范（常量或非空对象调用）
     - 包装类对象值比较使用 equals 而非 ==
   - ✅ 空值安全检查
     - Comparator 比较前过滤 null 值
     - Optional 链式调用的空值处理
     - 集合操作前的空值判断
   - ✅ 类型安全检查
     - 强制类型转换前使用 instanceof 检查
     - 泛型使用规范
   - ✅ 集合处理规范
     - 禁止使用 Arrays.asList()、List.of()、Map.of()
     - Comparator 三个条件的满足性
   - ✅ 命名规范检查
     - 驼峰命名、常量命名
     - 避免拼音与英文混合
   - ✅ 日志规约检查
     - PerfDuration 和 TraceDuration 名称与方法名一致
   
   **第二步：架构设计分析（深度必做）**
   - ✅ 职责清晰性检查（特别针对新增代码）
     - 方法是否承载了过多职责（单一职责原则）
     - application 层是否包含应该下沉到 domain 层的业务逻辑
     - 复杂方法是否需要拆分为多个小方法（建议 < 50 行）
     - 类的职责是否明确，是否存在"上帝类"
   - ✅ 层次分离检查
     - Application 层、Domain 层、Infrastructure 层职责是否清晰
     - 是否存在跨层调用
   - ✅ 设计一致性检查
     - 新增的设计模式是否与现有代码风格一致
     - 接口设计是否统一
   - ✅ 向后兼容性检查
     - 删除的 RPC 方法是否影响其他服务
     - API 变更是否考虑兼容性
   
   **第三步：质量、性能、安全问题**
   - 识别潜在的性能问题（循环中的数据库操作、重复计算等）
   - 识别安全风险
   - 提供具体的改进建议

   **⚠️ 重要提醒**：
   - 编码规范检查和架构设计分析**同样重要**，不可偏废
   - 必须两者都做完整，不能只做其中一项
   - 审查报告必须包含两个维度的问题

4. 生成审查报告
   - 使用 `create_file` 工具直接创建 `<仓库名>_PR<PR_ID>_<时间戳>/<仓库名>_<PR_ID>_<时间戳>_review.md`
   - **重要**：审查报告要保存在与 diff 和 info 文件相同的文件夹中
   - **重要**：直接生成 markdown 文件即可，无需使用 PowerShell 命令
   - 确保使用 UTF-8 编码，避免中文乱码
   - 包含详细的问题分析和修复建议

## 💡 AI 审查流程（详细）

### 准备阶段

```
用户提供: PR URL
AI 执行: 带时间戳的文件夹，包含 diff.txt 和 info.json 文件
示例文件夹: sp-search-citsgbt_PR174_202512261133/<PR_URL>
确认: 生成了 diff.txt 和 info.json 文件
```

### 分析阶段

**1. 加载所有上下文**
```
读取: 编码规范.mPR<PR_ID>_<时间戳>/<仓库名>_<PR_ID>_<时间戳>_info.json (PR元信息)
读取: <仓库名>_PR<PR_ID>_<时间戳>/<仓库名>_<PR_ID>_<时间戳内容)  
读取: <仓库名>_<PR_ID>_info.json (PR元信息)
读取: <仓库名>_<PR_ID>_diff.txt (完整diff)
```

**2. 理解 PR 背景**
- PR 的目的是什么？
- 涉及哪些模块？
- 是新功能、bug修复还是重构？

**3. 逐文件审查**

对于 diff 中的每个文件：

a) **识别文件类型**
   - Java 源文件
   - 配置文件
   - POM 文件
   - 测试文件

b) **过滤测试文件**
   - ⚠️ **重要**: 跳过所有测试文件的审查（`src/test/` 目录下的所有文件）
   - 测试代码不需要进行编码规范审查
   - 仅审查生产代码（`src/main/` 目录下的文件）

c) **分析变更内容**
   - 新增了什么？
   - 删除了什么？
   - 修改了什么？

d) **应用规范检查（按顺序）**

   **第一阶段：编码规范检查**（参考 CODE-STANDARDS.md）
   - ✅ 命名规范（类名、方法名、变量名、常量名）
   - ✅ 常量定义（是否有魔法值，包括中文字符串）
   - ✅ OOP规约（equals、@Override、静态访问等）
   - ✅ 集合处理（hashCode/equals、禁用方法、Comparator）
   - ✅ 日志规约（PerfDuration、TraceDuration）
   - ✅ 开关配置（命名规则）
   - ✅ BigDecimal使用（compareTo、构造函数）
   - ✅ CR中体现的问题（循环中操作DB、@Transactional、嵌套循环）

   **第二阶段：错误码规范检查**（参考 ERROR-CODE-STANDARDS.md）
   - ✅ ErrorCategory 使用是否正确
   - ✅ ErrorCode 命名是否符合规范
   - ✅ ErrorMsg 格式是否正确
   - ✅ ErrorSource 定义是否合理

   **第三阶段：架构设计分析**
   - ✅ 职责清晰性（单一职责、层次分离、方法复杂度）
   - ✅ 设计一致性（与现有代码风格是否一致）
   - ✅ 向后兼容性（RPC方法删除、API变更）
   - ✅ 扩展性考虑（是否便于未来扩展）

e) **质量分析

e) **职责清晰性分析**（重点关注新增代码）
   - **单一职责原则检查**
     - 方法是否只做一件事？
     - 方法代码行数是否过长（建议不超过 50 行）
     - 方法是否包含多个层次的逻辑（如既有业务逻辑又有数据访问）
   
   - **分层架构检查**
     - Application 层：是否只负责协调和流程控制？
     - Domain 层：核心业务逻辑是否在 domain 层？
     - Infrastructure 层：技术细节是否被正确封装？
     - 是否存在跨层调用或层次混乱？
   
   - **方法拆分建议**
     - 识别可以提取的独立逻辑块
     - 建议拆分为独立的私有方法或辅助类
     - 提供具体的拆分方案示例
   
   - **上帝类识别**
     - 类的方法数量是否过多
     - 类是否承担了过多不相关的职责
     - 建议拆分的方向和理由**
   - 代码重复
   - 复杂度
   - 性能问题
   - 安全隐患
   - 可维护性

**4. 综合评估**
   - 统计问题数量和严重程度
   - 评估整体代码质量
   - 给出审查结论

## 🐛 常见问题

### Q1: 为什么中文注释显示为乱码（如 `æ æ¿éåº`）？

**A**: 这不是代码问题，是 AI 读取文件时的字符集解析问题。实际代码中的中文是正常的。

**示例**:
- AI 读取显示: `"æ æ¿éåº"`
- 实际代码内容: `"无房过滤"`

**处理方式**:
- ✅ **正确**: 理解这是 UTF-8 字节序列，在报告中使用正确的中文
- ❌ **错误**: 将其作为"注释编码问题"报告给用户

### Q2: 中文字符串字面量算不算魔法值？

**A**: 算！即使是正确显示的中文字符串（如 `"无房过滤"`、`"不符合直连优先"`），根据编码规范，仍然属于魔法值，应该定义为常量。

**原因**:
- 便于维护（统一修改）
- 支持国际化（i18n）
- 提高代码可读性

### Q3: 如何处理测试文件？

**A**: 跳过所有 `src/test/` 目录下的文件审查。测试代码可以使用一些在生产代码中禁止的写法（如 `List.of()`、魔法值等）。

### Q4: 工具执行失败怎么办？

**常见错误**:

**401 认证错误**
- 原因: API Token 无效或过期
- 解决: 重新生成 token，确认 username 是邮箱地址

**404 错误**
- 原因: PR 不存在或无权限访问
- 解决: 检查 PR URL，确认访问权限

**找不到规范文件**
- 原因: 当前目录缺少规范文档
- 解决: 确保 `CODE-STANDARDS.md` 和 `ERROR-CODE-STANDARDS.md` 在当前目录

### Q5: 如何判断问题的严重程度？

**严重程度分级**:

**🔴 高优先级 (CRITICAL)** - 必须修复后才能合并
- 违反【强制】规范
- 严重影响代码质量、性能或安全性
- 可能导致运行时错误
- 向后兼容性破坏（RPC/API 删除）

**🟡 中优先级 (IMPORTANT)** - 强烈建议修复
- 违反【建议】规范
- 影响代码可维护性
- 可能引入潜在问题

**🟣 架构和设计问题 (DESIGN)** - 需要重构
- 职责不清晰（违反单一职责原则）
- 层次划分不清晰
- 设计不一致
- 方法过长需要拆分

**🟢 低优先级 (MINOR)** - 可以合并后修复
- 代码风格问题
- 优化建议
- 注释和文档
- @Override 注解缺失（不影响功能）

### Q6: 审查时应该先看什么？

**A**: 必须按照以下顺序进行审查，**两者同样重要，不可偏废**：

1. **第一步：编码规范检查**（基础质量）
   - 魔法值、@Override、空值检查、类型安全等
   - 这是代码质量的基础，不可跳过
   
2. **第二步：架构设计分析**（深度质量）
   - 职责清晰性、层次分离、设计一致性
   - 发现更深层次的架构问题

**❌ 错误示例**：只做架构分析，忽略编码规范
**✅ 正确示例**：同时完成编码规范检查和架构设计分析

**为什么两者都重要**：
- 编码规范保证代码的**表层质量**（可读、可维护）
- 架构设计保证代码的**深层质量**（可扩展、可测试）
- 缺少任何一项都不是完整的代码审查
- 不影响功能的改进

### Q7: 如何避免审查时的误判？

**A**: 在审查新增代码时，务必遵循以下原则：

**⚠️ 审查新增代码的正确方法**：

1. **仔细阅读 diff 文件中的新增代码**
   - diff 文件中以 `+` 开头的行是新增代码
   - 确保理解每一行新增代码的内容
   - 不要凭想象或假设代码内容

2. **检查前需确认代码实际内容**
   - ❌ 错误：假设某个注解或代码不存在
   - ✅ 正确：查看 diff 确认该代码是否真的缺失
   
3. **常见误判案例**：
   ```diff
   + @Override
   + public int getOrder() {
   +     return HotelAvailMarker.ORDER - 100;
   + }
   ```
   - ❌ 误判：指出"缺少 @Override 注解"
   - ✅ 正确：代码已经包含 @Override 注解，无需报告

4. **审查检查清单**：
   - [ ] 我已经查看了 diff 文件中的实际代码
   - [ ] 我确认了问题确实存在
   - [ ] 我没有基于假设进行判断
   - [ ] 对于复杂情况，我已经多次确认

**真实案例警示**（v3.2.0）：
- **错误场景**：在审查 PR #174 时，错误指出 PayLabelPreferredMarker.java 缺少 @Override 注解
- **实际情况**：通过 diff 文件可以清楚看到，代码在第21行和第26行都正确添加了 @Override 注解
- **教训**：必须基于实际代码内容进行审查，不能凭印象或猜测
- **改进**：审查前先完整阅读 diff 文件，确认问题真实存在后再报告

### 🔥 v3.2.0 更新说明

**本次更新是一次重大方法论更新**，旨在防止 AI 在审查时遗漏基础编码规范检查。

- 编码规范是代码质量的基础

1. ✅ 先完整检查编码规范（魔法值、注解、命名等）
2. ✅ 再深入分析架构设计（职责分离、层次划分等）
3. ✅ 生成包含两类问题的完整报告

**错误做法**：
- ❌ 只关注架构设计，忽略编码规范
- ❌ 认为架构分析可以替代规范检查
- ❌ 分两次审查（应该一次完成双重检查）
